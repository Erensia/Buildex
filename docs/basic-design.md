# Buildex 기본 설계

## 기술 구성

- Next.js App Router, TypeScript, React, Tailwind CSS
- PostgreSQL 16, Drizzle ORM
- Auth.js Credentials 인증
- Zod 입력 검증, Vitest 단위 테스트

## 주요 흐름

1. 비회원은 `/characters`에서 현재 공개 릴리스의 캐릭터·무기·에코 세트와 운영 예시 빌드 결과를 읽는다.
2. 로그인한 사용자는 `/build`에서 캐릭터, 호환 무기, 에코 세트, 주옵션 및 부옵션을 선택한다.
3. 화면은 선택 내용을 바탕으로 즉시 스탯을 계산한다.
4. 저장 또는 수정 시 서버가 입력을 검증하고 동일한 계산식을 적용해 `build_profiles`에 기록한다.
5. 회원은 저장 빌드를 즐겨찾기에 등록하고, 즐겨찾기 또는 저장 빌드를 3개 파티 슬롯에 배치해 조합을 관리한다. 이 기능은 대미지를 계산하지 않는다.
6. 저장 목록에서 빌드를 불러와 수정하거나 삭제하고, 결과 카드에 표시되는 모든 지원 스탯을 다른 저장 빌드와 비교할 수 있다.

## 파티 최종 스탯 계산 흐름

1. 사용자는 메인 빌드 캐릭터와 최대 2명의 지원 캐릭터를 편성한다.
2. 계산기는 메인 캐릭터, 지원 캐릭터와 사용자가 확인한 조건을 기준으로 적용 가능한 파티 버프를 판정한다.
3. 파티 버프는 게임 데이터 릴리스에 귀속한다. 적용된 버프만 메인 빌드의 최종 스탯에 합산하고, 각 버프의 제공자·조건·편성 충족 여부·최종 반영 여부를 결과에 표시한다.
4. 이 단계는 DPS를 추정하지 않는다. 스킬·적·시간축 데이터가 충분히 검증된 뒤 전체 로테이션 DPS 시뮬레이션을 별도 기능으로 추가한다.

## 관리자 데이터 운영 흐름

1. `ADMIN_EMAIL`과 일치하는 계정이 로그인하면 서버가 해당 사용자를 `admin` 역할로 승격한다.
2. 관리자는 현재 공개 릴리스를 복제해 새 `draft`를 만들고, JSON 입력으로 초안 데이터만 편집한다. 공개·과거 릴리스는 읽기 전용이다.
3. 발행 전 `/api/admin/game-data/releases`가 릴리스 출처, 모든 행의 버전·검증일, 캐릭터별 호환 무기, 1·3·4 코스트 주옵션 및 모든 에코·세트의 소속 관계를 검사한다.
4. Publish는 새 초안을 `published`로, 기존 공개본을 `superseded`로 전환하고 게임의 현재 릴리스 포인터를 같은 트랜잭션에서 갱신한다.
5. 플래너와 저장 API는 기본적으로 현재 공개 릴리스를 읽는다. 저장 빌드를 열면 해당 빌드의 공개·보존 릴리스를 다시 읽고, 수정도 같은 릴리스를 기준으로 검증한다. 최신 패치로 옮길 때는 사용자가 현재 공개 데이터로 복제한 뒤 명시적으로 저장한다.
6. 플래너 데이터 API는 선택된 릴리스의 `sourceSnapshot`과 `sourceManifest`를 함께 전달한다. 화면은 게임 행에 남아 있을 수 있는 대표 URL이 아니라 릴리스 출처 목록을 표시한다.
7. 저장 빌드의 `formula_version`이 현재 번들과 다르면 결과 패널은 `calculated_result` 스냅샷을 재현한다. 사용자가 현재 계산식으로 재계산을 명시적으로 실행한 경우에만 서버가 현재 번들 결과와 버전을 저장한다.
8. `build_favorites`는 사용자와 저장 빌드의 관계만 보관하며, 즐겨찾기 대상은 반드시 사용자가 소유한 빌드여야 한다. `party_profiles`는 사용자별 이름과 정확히 세 개의 빌드 슬롯을 보관한다. 모든 API는 사용자 소유권을 서버에서 검사한다.

## 범위 제한

- 소셜 로그인은 이번 작업 범위에서 제외한다.
- 파티 DPS·로테이션 시뮬레이션은 게임 공식의 검증 범위가 확보될 때까지 구현하지 않는다. 현재 파티 기능은 3인 구성 관리에 한정한다.
- 등급 판정은 현재 창리·자니·히유키의 S0 기준을 제공한다. 다른 캐릭터의 등급 기준과 범용 추천은 이후 확장한다.

## 이후 고도화 목표

- 지원 캐릭터와 캐릭터별 등급 기준을 확대한다.
- 계산식과 조건부 효과의 정확도를 출처 기반으로 검증한다.
- 릴리스 검증, 출처 관리, 운영 도구를 게임 데이터 갱신 규모에 맞춰 고도화한다.
