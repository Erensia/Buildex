# 게임 데이터 갱신 기준

게임 데이터는 서버 계산의 단일 기준이며, 행 단위의 현재 값이 아니라 발행된 **데이터 릴리스**로 관리한다. 패치가 적용되면 캐릭터, 무기, 에코, 에코 세트, 주옵션을 같은 릴리스에 담아 발행한다.

## 릴리스 원칙

- `game_data_releases`는 게임·버전·검증일·출처 묶음·상태를 가진 불변 스냅샷이다. 상태는 `draft`, `published`, `superseded`를 사용한다.
- 공개 API와 플래너는 게임의 `current_data_release_id`가 가리키는 `published` 릴리스만 읽는다. 초안은 사용자에게 노출하지 않는다.
- 캐릭터·무기·에코·세트·주옵션은 `release_id`에 귀속된다. 같은 외부 식별자는 서로 다른 릴리스에 공존할 수 있어 과거 빌드를 재현할 수 있다.
- 빌드 저장 시 서버가 계산에 사용한 `data_release_id`를 함께 저장한다. 과거 빌드는 당시 기준으로 보존하고, 사용자가 원할 때만 현재 릴리스로 복사·재계산한다.
- 발행은 검증을 통과한 초안을 하나의 트랜잭션으로 전환한다. 이전 공개 릴리스는 `superseded`로 보존한다.

## 관리자 워크플로

1. 관리자는 릴리스 대시보드에서 현재 `published` 릴리스를 확인하고, 이를 복제해 새 버전의 `draft`를 만든다. 초안 생성 시 버전, 검증 날짜, 대표 출처 묶음과 변경 메모를 기록한다.
2. JSON 편집기는 유지하되 데이터 저장 대상은 선택한 `draft`뿐이다. `published`와 `superseded` 릴리스는 읽기 전용이며 직접 수정·삭제할 수 없다.
3. `Publish` 전에 서버가 데이터 행의 버전·검증일 일관성, 캐릭터·무기·에코·세트·주옵션의 최소 구성, 에코 세트 소속 관계를 검사한다.
4. 검증을 통과하고 관리자가 확인하면 한 트랜잭션에서 새 초안을 `published`로, 이전 공개 릴리스를 `superseded`로 변경하며 게임의 현재 릴리스 포인터를 전환한다.
5. 오류 수정은 공개본을 덮어쓰지 않고 새 hotfix 초안을 만들어 발행한다. 저장된 빌드는 기존 `data_release_id`를 유지한다.

## 갱신 절차

1. 공식 패치 노트 또는 검증 가능한 게임 데이터 출처를 기록하고, 출처 묶음과 변경 요약을 포함한 초안을 만든다.
2. 초안에 캐릭터·무기·에코·세트·주옵션을 같은 버전과 검증일로 등록한다. 대량 초기 적재나 스키마 변경만 마이그레이션을 사용한다.
3. 무기 타입, 에코 비용·소속, 주옵션, 현재 지원 계산 규칙을 검증한 뒤 발행한다.
4. 발행 시 `games.current_data_release_id`와 표시용 버전을 같은 트랜잭션에서 전환한다.
5. 계산·입력 검증·빌드 데이터 API가 새 릴리스만 읽고, 저장 빌드가 사용한 릴리스를 유지하는지 테스트한다.

플래너는 현재 사용 중인 데이터 버전과 검증 날짜, 출처 링크를 화면에 표시한다. 이로써 저장된 빌드의 `data_version`과 현재 게임 데이터를 비교할 수 있다.

관리자 계정은 환경 변수 `ADMIN_EMAIL`로 지정한다. 이 이메일로 로그인한 사용자는 최초 인증 요청 때만 자동으로 `admin` 역할이 되며, 다른 회원가입 계정은 관리자 권한을 얻지 못한다.
