## 목표와 우선순위

이번 단계의 목표는 단순히 목록 수를 늘리는 것이 아니다. 새 캐릭터·무기·에코 데이터가 추가되어도 계산값, 호환 무기 목록, 저장 빌드의 재현성이 함께 유지되는 운영 기반을 만드는 것이다.

작업은 아래 순서로 진행한다.

1. 계산 정확도 검증 기반을 먼저 고정한다.
2. 검증된 형식으로 지원 캐릭터와 연관 데이터를 확장한다.
3. 릴리스 검증·변경 추적·관리자 운영 흐름을 고도화한다.

캐릭터 수만 먼저 늘리면 잘못된 입력과 계산식을 더 넓게 전파하게 된다. 보기에는 그럴듯해도 계산기는 숫자를 틀리면 장식품이므로, 이 순서를 따른다.

## 1. 지원 캐릭터 확대

### 목적

- 현재 공개 캐릭터 라이브러리와 빌드 플래너가 더 많은 지원 캐릭터를 일관되게 제공한다.
- 캐릭터의 무기 타입, 역할, 기초 스탯, 호환 무기와 파티 편성 정보가 같은 릴리스 기준으로 연결된다.

### 작업 범위

- 추가 대상 캐릭터와 각 캐릭터의 `externalKey`, 한글명, 역할, 무기 타입, 기초 스탯을 확정한다.
- 해당 무기 타입의 무기, 에코·에코 세트·주옵션 데이터가 릴리스 내에서 빠지지 않도록 함께 보강한다.
- 공개 `/characters` 필터와 `/build` 선택 UI가 신규 데이터를 같은 방식으로 표시하는지 확인한다.
- 3인 파티 편성은 멤버 선택·저장 기능만 제공하며, 파티 DPS나 로테이션 계산은 범위에서 제외한다.

### 확장 대상 (적용 완료)

현재 지원 3파티(용융: 장리·루파·브렌트 / 회절: 젠니·페비·벨리나 / 응결: 히유키·치사·루실라)에 이어, 남은 속성마다 다음의 대표 3인 파티를 추가 후보로 둔다.

| 속성 | 후보 파티 | 의도 |
| --- | --- | --- |
| 기류 | 기염 · 모르테피 · 벨리나 | 기류 메인 딜러와 공용 서포터 편성을 검증 |
| 전도 | 상리요 · 음림 · 벨리나 | 전도 메인·서브 딜러와 공용 서포터 편성을 검증 |
| 인멸 | 카멜리아 · 로코코 · 파수인 | 인멸 메인·서브 딜러와 파수인 편성을 검증 |

- 별도 기류 편성으로 **유노 · 감심 · 파수인**을 지원한다.
- 인멸 지원 캐릭터에 **양양·현령**을 추가한다.

각 캐릭터는 한글명, 역할, 속성, 무기 타입, 레벨 90 기초 공격력, 출처 URL을 확정한다. 플래너에 필요한 무기 데이터는 기존 호환 무기를 우선 사용하고, 각 파티의 전용 무기는 검증 가능한 기초 공격력·주옵션·출처가 확보될 때 별도 추가한다.

### 데이터 입력 방식 (구현 완료)

관리자 화면의 한 항목씩 JSON 입력은 9명의 캐릭터와 무기를 검수하기에 오류 가능성이 높다. 따라서 아래의 **파티 번들 초안 입력**을 추가한다.

1. 관리자가 현재 공개 릴리스에서 새 `draft`를 만든다.
2. 속성별 JSON 템플릿에 캐릭터 **정확히 3명**과 선택 무기를 채운다. 각 행은 `externalKey`, 한글명, 역할, `element`, `weaponType`, `baseAttack`, `sourceUrl`을 필수로 갖는다.
3. 서버는 저장 전에 키 중복, 3인 파티 완결성, 무기 타입, 스탯 숫자·출처 URL을 검사하고 추가·변경 요약을 미리 보여 준다.
4. 검수자가 미리보기 결과를 확인한 뒤에만 초안에 반영한다. `Publish` 전 기존 릴리스 검증을 다시 수행한다.

이 방식은 사용자가 원본 데이터를 직접 채워도 입력 형식을 강제하고, 구현자는 출처·타입·연결 관계만 집중 검수할 수 있게 한다.

### 완료 기준

- 추가 캐릭터마다 호환 무기 목록이 해당 무기 타입 전체와 정확히 일치한다.
- 캐릭터 카드와 플래너에서 무기 타입·역할·기초 스탯 정보가 깨지지 않는다.
- 릴리스 검증이 캐릭터와 호환 무기 부재를 차단한다.
- 각 추가 캐릭터에 대해 공개 조회와 빌드 입력 검증을 자동 테스트한다.

## 2. 계산 정확도 검증

### 목적

- 표시 스탯의 출처와 적용 순서를 명시하고, 에코 주옵션·부옵션·무기·활성 버프가 즉시 합산되는지 보장한다.
- 지원하지 않는 효과를 계산한 것처럼 보이지 않게 분리한다.

### 계산 계약

| 항목 | 현재 단계의 처리 원칙 |
| --- | --- |
| 공격력 | `(캐릭터 기초 공격력 + 무기 기초 공격력) × (1 + 공격력%) + 고정 공격력` |
| 치명타 확률 | 기본 5% + 장비·에코·활성 버프 기여치 |
| 치명타 피해 | 기본 150% + 장비·에코·활성 버프 기여치 |
| 공명 효율 | 기본 100% + 장비·에코·활성 버프 기여치 |
| 속성·스킬 피해 보너스 | 관련 장비·에코·활성 버프 기여치의 합 |
| 파티 효과 | 데이터로 선언된 조건부 기여치만 사용자가 활성화했을 때 합산 |
| DPS·회전·확률 기대값 | 이번 범위에서 계산하지 않으며, UI와 문서에 비계산 항목으로 유지 |

### 작업 범위

- 계산 모듈의 중복 기본 스탯 선언과 릴리스 검증 API의 중복 검사를 제거하기 전에 회귀 테스트를 추가한다.
- `StatValues`의 지원 키, 단위(퍼센트/고정값), 적용 위치를 단일 계약으로 문서화하고 런타임 검증에 연결한다.
- 에코 부옵션의 허용 롤·중복 금지·합산 결과를 단위 테스트와 대표 빌드(golden case)로 검증한다.
- 공격력, 치명타, 공명 효율, 속성 피해, 조건부 파티 버프의 조합 케이스를 테스트한다.
- 결과 화면의 표기 단위와 반올림 규칙을 테스트 기준과 맞춘다.

### 완료 기준

- 기본 스탯이 데이터 존재 여부와 무관하게 정확히 한 번만 적용된다.
- 에코 부옵션을 바꾸면 결과가 입력 직후 기대값만큼 변한다.
- 모든 지원 스탯 키가 유효성 검사·계산·표시 중 하나에서 누락되지 않는다.
- 알려진 대표 빌드의 계산 결과를 고정 테스트로 재현한다.

## 3. 데이터 운영 고도화

### 목적

- 게임 패치 데이터의 출처, 검증, 차이, 발행 이력을 추적 가능하게 만든다.
- 관리자 JSON 입력 실수나 서로 다른 릴리스 데이터의 혼입을 공개 전에 차단한다.

### 작업 범위

- 릴리스 검증을 캐릭터·무기 타입 호환성, JSON 스키마, 스탯 키/숫자 범위, 에코 세트 관계, 파티 버프 참조까지 확대한다.
- 초안과 현재 공개 릴리스의 캐릭터·무기·에코·주옵션 변경 요약(diff)을 관리자에게 제공한다.
- 출처 묶음(`sourceManifest`), 검증 날짜, 변경 메모를 릴리스 발행 전 필수 확인 항목으로 만든다.
- 발행 전 데이터 품질 검사와 발행 후 공개 API 스모크 테스트를 자동화한다.
- 마이그레이션 번호·저널·스키마의 일관성을 점검하고, 새 데이터 변경은 재현 가능한 마이그레이션으로 남긴다.

### 완료 기준

- 잘못된 JSON 형태, 미지원 스탯 키, 타입 불일치, 끊어진 참조는 발행 전에 오류로 표시된다.
- 관리자는 새 초안에서 이전 공개본 대비 추가·변경·삭제 대상을 확인할 수 있다.
- 공개 데이터는 항상 하나의 `published` 릴리스에서만 읽히며, 저장 빌드는 자신의 `data_release_id`를 보존한다.
- 데이터 변경은 출처와 검증 결과를 남기며, 릴리스 검증 테스트를 통과해야 발행할 수 있다.

## 진행 기록

### A. 계산 정확도 기준선 — 완료 (2026-07-22)

- 프로덕션 빌드를 막던 중복 선언을 정리했다. 대상은 빌드 데이터 API, 관리자 릴리스 검증 API, 빌드 플래너, 계산기 기본 스탯 필드다.
- 기본 치명타 확률 5%, 치명타 피해 150%, 공명 효율 100%를 `STANDARD_BASE_STATS`로 단일화해 정확히 한 번만 더한다.
- 스탯 기여치를 덮어쓰지 않고 합산하는 `addStats`를 도입했다. 특히 동일 에코의 주옵션과 부옵션이 같은 스탯일 때 두 값이 함께 반영된다.
- 해당 에코 합산 회귀 테스트를 추가했고, `pnpm lint`, `pnpm test`(23개), `pnpm build`를 통과했다.

### B. 지원 캐릭터 확대 — 완료 (2026-07-22)

- 사용자 확정 파티를 기류(기염·모르테피·벨리나), 전도(상리요·음림·벨리나), 인멸(카멜리아·로코코·파수인)로 변경했다.
- 기류 후보에 유노, 인멸 후보에 양양·현령을 추가했다.
- 관리자 초안에 **정확히 3명**을 받는 파티 번들 JSON 입력과 추가·갱신 미리보기를 추가했다. 4명 이상·중복 멤버는 검증 단계에서 거부한다.
- `3.5.1` 공개 릴리스를 발행하여 기염, 모르테피, 벨리나, 유노, 감심, 상리요, 음림, 카멜리아, 로코코, 파수인, 양양·현령을 반영했다. 파수인은 범용 표기만 사용하고, 양양·현령은 지원 역할을 포함하지 않는다.
- 권총·권갑 호환성을 위해 정적의 안개, 심연의 파도를 보강했고, 기류·전도·인멸 피해 보너스 3코스트 주옵션을 계산·표시·검증 경로에 추가했다.
- 기존 치사의 문자열형 JSON 기초 스탯도 객체형 JSON으로 정규화해, 호환 무기와 플래너가 같은 형식으로 읽도록 보정했다.
- 이전 `3.5` 릴리스는 `superseded`로 보존하여 저장 빌드가 기존 데이터 릴리스를 계속 참조한다.
- DB 적용 후 캐릭터 19명·무기 12개를 대상으로 호환 무기 부재가 0건인지, 릴리스 메타데이터가 모두 `3.5.1`/`2026-07-22`인지 확인했다.

### C. 데이터 운영 고도화 — 진행 중 (2026-07-22)

- 입력 스키마가 캐릭터의 기초 공격력·속성·무기 타입, 무기·에코 스탯의 지원 키와 유한 수치, 에코 주옵션 키를 저장 전에 검증하도록 강화했다.
- 릴리스 발행 검증도 위 조건을 다시 확인하고, 호환 무기·에코 세트 연결·파티 버프 참조·릴리스 메타데이터를 함께 점검한다.
- 관리자 초안에서 현재 공개 릴리스 대비 캐릭터·무기·에코·에코 세트·주옵션의 추가·변경·삭제 항목을 확인하는 diff를 추가했다.
- 발행 트랜잭션 안에서 게임의 공개 릴리스 포인터와 릴리스 상태를 재조회하는 스모크 검증을 추가했다.

## 범위 밖과 승인 지점

- 정확한 실전 DPS, 로테이션, 스킬 계수, 몬스터 방어력 기반 시뮬레이션은 제외한다. 공식이 불완전하거나 조건이 많은 영역을 추측으로 자동화하지 않는다.
- 신규 캐릭터의 실제 수치·패치 버전·출처 선택은 데이터 적재 직전에 공식 자료와 대조하고, 불확실한 값은 사용자 확인을 요청한다.
- 데이터 모델을 크게 바꾸거나 과거 저장 빌드의 재계산 의미가 달라질 경우, 마이그레이션 적용 전에 사용자 승인을 받는다.

## 공통 품질 게이트

각 단계가 끝날 때 다음을 수행하고 `docs/development-log.md`에 결과를 기록한다.

1. 스키마·마이그레이션과 공개 API의 릴리스 일관성 확인
2. `pnpm lint`, `pnpm test`, `pnpm build` 실행
3. 변경된 데이터의 출처·버전·검증일 확인
4. 공개 화면과 회원 전용 저장/즐겨찾기/파티 편성에 대한 회귀 확인
